z.ai GLM API 踩坑记录
Pitfall 2026-05-02 修改于 2 小时前 编辑 ↗

问题背景

使用 z.ai GLM 系列模型 API 时，遇到了一系列非预期行为。记录于此，避免重复踩坑。

坑 1：Smart Routing 静默降级

当启用 smart_routing 参数时，API 会静默将复杂请求降级到 GLM-4.5-Air 而不通知调用方。这导致：

  • 输出质量明显下降，但 HTTP 200 正常返回
  • 响应头 x-model-used 才能看到实际使用的模型
  • 必须检查此 header 才能确认是否被降级
关键教训：永远不要信任 API 会按请求的模型返回结果，务必检查响应头。

坑 2：Token 计数不一致

z.ai 的 token 计数与 OpenAI 标准存在差异，中文文本差异尤其明显：

# OpenAI 标准
prompt_tokens: 1247
completion_tokens: 512

# z.ai 实际
prompt_tokens: 1583  # 多 ~27%
completion_tokens: 641  # 多 ~25%

影响

如果你按 token 数做成本估算或上下文窗口管理，必须用 z.ai 自己的计数，不能套 OpenAI 的经验值。

坑 3：并发限制无文档

官方文档写着 Max Plan 支持"高并发"，但实际测试发现：

  • 同时发 3 个请求正常
  • 第 4 个请求直接 429 Too Many Requests
  • 并发限制 = 3，但文档里没写

解决方案

针对以上问题，目前的应对策略：

  1. 每次请求后检查 x-model-used 响应头
  2. 用 z.ai 的计数器做 token 管理
  3. 控制并发 ≤ 3，或加请求队列
